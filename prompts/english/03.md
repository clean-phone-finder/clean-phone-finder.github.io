# Smartphone Prompt Generator — Build Specification (English)

Create an HTML file in English — a fully static single-page web application with a step-by-step wizard interface (4 screens) for generating a text prompt to help choose a smartphone free of intrusive features.

---

## HEAD & SEO

```html
<title>Smartphone Without Bloatware, Ads & Spyware — AI Prompt Generator</title>
<meta name="description" content="Find smartphones without ads, bloatware, spyware, and overheating. Generate an AI research prompt based on real Reddit reviews and user reports.">
<meta charset="UTF-8">
<meta name="viewport" content="width=device-width, initial-scale=1.0">
<meta name="robots" content="index, follow">
<html lang="en">
<link rel="canonical" href="">
```

No extended SEO or structured data.

---

## TECHNICAL REQUIREMENTS

The file must be a single self-contained HTML file with embedded CSS and JavaScript. No CSS frameworks, JavaScript frameworks, libraries, or CDNs are allowed. The page must be fully static and responsive. SVG icons for AI assistants must render correctly in both themes.

---

When these JSON objects are changed, the page must automatically reflect the updated data without modifying any other part of the code.

Each classifier (phones, countries, languages) has two lists. The first — `chips` — contains items displayed as chip buttons. The second — `autocomplete` — contains the full expanded list of all known values for that classifier, including those already in `chips`. The Other field for each classifier uses the `autocomplete` list for suggestions via HTML datalist. ## QUERY PARAMS & STATE

The state of all selected values is stored in URL query params. The `phone` parameter may contain one or more values separated by commas — e.g. `phone=iPhone+16,Pixel+9`. The `country` and `lang` parameters contain a single value or are absent. All parameters are optional. On page reload, state is restored from query params and the corresponding screen is shown. If no query params are present — screen 1 is shown.

---

## DESIGN SYSTEM

All colors, spacing, and typography are defined as CSS custom properties in the `:root` block. Hardcoded color values are not allowed anywhere in CSS except in `:root` and `[data-theme="dark"]`. Theme switching is done via the `data-theme="dark"` attribute on the `<html>` element. Theme state is stored in `localStorage`. The theme toggle is positioned in the top-right corner on every screen.

```css
:root {
  --bg: #ffffff;
  --surface: #f5f5f5;
  --border: #e0e0e0;
  --text-primary: #111111;
  --text-secondary: #666666;
  --accent: #2563eb;
  --accent-hover: #1d4ed8;
  --accent-text: #ffffff;
  --radius: 8px;
  --font: system-ui, -apple-system, BlinkMacSystemFont, sans-serif;
}

[data-theme="dark"] {
  --bg: #0f0f0f;
  --surface: #1a1a1a;
  --border: #2a2a2a;
  --text-primary: #f0f0f0;
  --text-secondary: #999999;
  --accent: #3b82f6;
  --accent-hover: #2563eb;
  --accent-text: #ffffff;
}
```

Typography: `font-family` — `var(--font)`, base font size — 16px, `line-height` — 1.6. `h1` — 1.75rem, font-weight 600. Subtitle — 1rem, font-weight 400, color `var(--text-secondary)`. Buttons — 0.875rem, font-weight 500. Textarea — 0.9rem, line-height 1.7, font-family monospace.

Chip buttons use `var(--surface)` as background, `var(--border)` as border, `var(--text-primary)` as text color. An active chip uses `var(--accent)` as background and `var(--accent-text)` as text color. All interactive elements use `var(--radius)` for border-radius. Transitions on color and background-color — 0.2s ease. No box-shadow anywhere except a thin 1px border on surfaces via `var(--border)`.

Progress bar at the top of each screen, 2px height, color `var(--accent)`, filled according to the current step: screen 1 — 25%, screen 2 — 50%, screen 3 — 75%, screen 4 — 100%.

## SCREEN 1 — SMARTPHONE SELECTION

`h1` at the top: "Find a Smartphone Without Ads, Bloatware & Spyware". Subtitle below: "Generate a research prompt for AI based on real Reddit reviews and user reports". Step indicator: "STEP 1 OF 4 · SMARTPHONE". Hint below the indicator: "Select one to analyze or multiple to compare".

Chip buttons are generated from `CONFIG.phones.chips`. Multi-select is supported — any number of chips can be selected simultaneously. Clicking an active chip deselects it.

The last button is "Other". When "Other" is clicked, a row appears below the chips containing a text input field (with an HTML datalist populated from `CONFIG.phones.autocomplete`) and an "Add" button next to it. A model is added to the tag list either by clicking the "Add" button **or** by pressing Enter in the input field. After adding, the input is cleared and remains focused for the next entry. Each added tag is displayed below the input row with an × button to remove it. If the user attempts to add a model that already exists among the active chips or existing tags, the addition is silently ignored. Any number of tags can be added. Values entered via Other are merged with selected chips and all are stored together in the `phone` query param separated by commas — no separate parameter for Other exists. When restoring state from query params, values that do not match any chip in `CONFIG.phones.chips` are displayed as tags in the Other field, while the rest are shown as active chips.

A "Skip" button skips the step without any selection. A "Next" button proceeds to screen 2 and saves all selected values in the `phone` query param separated by commas.

---

## SCREEN 2 — COUNTRY SELECTION

Step indicator: "STEP 2 OF 4 · COUNTRY". Chip buttons are generated from `CONFIG.countries.chips`. Single select — only one country can be selected. The last button is "Other". When "Other" is clicked, the button becomes active (like a chip) and a text input field with an HTML datalist populated from `CONFIG.countries.autocomplete` appears below the chips. There is no Add button. If the user selects any other chip — the Other button is deactivated, the input field is hidden, and its content is cleared. When "Next" is clicked — if the Other button is active and the field is not empty — the value from the field is used as the selected country and saved in the `country` query param. If the Other button is active but the field is empty — the step is treated as "Skip" (no value is saved). A "Skip" button. A "Back" button returns to screen 1 while preserving state. A "Next" button proceeds to screen 3.

---

## SCREEN 3 — RESPONSE LANGUAGE SELECTION

Step indicator: "STEP 3 OF 4 · LANGUAGE". Chip buttons are generated from `CONFIG.languages.chips` — the `label` field is displayed. Single select. On screen load, the browser language is detected via `navigator.language` and matched against the `code` field in `CONFIG.languages.chips`. If a match is found — that button receives the label "(your browser language)" and is highlighted as recommended. If no match — no button is pre-highlighted. The last button is "Other". When "Other" is clicked, the button becomes active (like a chip) and a text input field with an HTML datalist populated from `CONFIG.languages.autocomplete` appears below the chips — the `label` field values are shown. There is no Add button. If the user selects any other chip — the Other button is deactivated, the input field is hidden, and its content is cleared. When "Next" is clicked — if the Other button is active and the field is not empty — the value from the field is used as the selected language and saved in the `lang` query param. If the Other button is active but the field is empty — the step is treated as "Skip" (no value is saved). A "Skip" button. A "Back" button returns to screen 2. A "Next" button proceeds to screen 4.

---

## SCREEN 4 — RESULT

Step indicator: "STEP 4 OF 4 · YOUR PROMPT". Summary of selected parameters: Smartphone (or Smartphones if more than one is selected), Country, Language. If a parameter was not selected — displays "Not specified". If multiple smartphones were selected — all are displayed separated by commas.

Below the summary, a read-only textarea with the generated prompt. The textarea automatically adjusts its height to fit the content.

A "Copy Prompt" button copies the textarea content to the clipboard and temporarily changes its label to "Copied!" for 2 seconds. A "Share" button is placed next to "Copy Prompt" and uses a secondary style (background `var(--surface)`, border `var(--border)`, text color `var(--text-primary)`). When "Share" is clicked, the presence of the `navigator.share` API is checked. If the API is available (primarily mobile devices) — the native share dialog is triggered with the current page URL including all query params. If the API is unavailable (desktop) — the current URL is copied to the clipboard and the button label temporarily changes to "Link Copied!" for 2 seconds, then returns to "Share". The "Share" button is always present on screen 4 — regardless of which parameters were filled in.

A "Start Over" button resets all query params and returns to screen 1. A "Back" button returns to screen 3.

Below the textarea — the AI assistants section, generated from `CONFIG.ai_assistants`. Below that — the links section, generated from `CONFIG.placeholder_links`. At the bottom — a link to GitHub from `CONFIG.github_url`.

## PROMPT GENERATION LOGIC

The generated prompt must be plain text with no emoji, no HTML, no Markdown, and no special formatting of any kind. The prompt is divided into logical paragraphs separated by blank lines for comfortable human reading. Every prompt variant must include a strict instruction for the AI not to fabricate facts, make unsourced assumptions, or present unverified information. The prompt must require reliance solely on real user reports, discussions, and public sources. Reddit is mandatory but not the only source. If a language is specified — the prompt must explicitly require the response to be in that language. If no language is specified — the response defaults to English.

The prompt must always include verification of the following aspects:

**Hardware issues:** overheating under load and during normal use, CPU and GPU throttling after sustained load, battery degradation, component instability, build quality issues.

**Software issues:** app freezes and crashes, unstable firmware updates, post-update bugs, performance degradation over time, RAM management and background process handling issues.

**Ads and privacy:** presence of built-in advertising in the system UI, spyware and adware in stock firmware, forced installation of apps without user consent, inability to disable or remove intrusive system apps, collection and transmission of user data without explicit consent.

**Sensors and hardware compromises:** presence of a virtual (software-based) proximity sensor instead of a dedicated hardware infrared sensor. The prompt must explicitly require checking whether the device uses software emulation of the proximity sensor via accelerometer or ambient light sensor instead of a dedicated hardware IR sensor, and if so — describe real user complaints about incorrect screen behavior during calls.

**Regional differences:** firmware differences between markets, regional feature restrictions, presence of additional bloatware in regional versions.

**Time on market:** if the model has been on the market for less than one year — the prompt must explicitly require warning the user about the insufficient number of independent long-term reviews.

**Alternatives:** the prompt must require suggesting specific alternative models without the described issues.

### Prompt variants depending on filled fields

**If nothing is specified (or only language):** a general worldwide smartphone selection prompt based on real user discussions.

**If only a country is specified (or country + language):** the prompt accounts for regional specifics, model availability, regional firmware, restrictions, and user complaints specific to that region.

**If a single model is specified (or one model + country, or one model + language, or all three fields with one model):** the prompt focuses on analyzing that specific model against all aspects listed above, taking the country into account if specified.

**If multiple models are specified:** the prompt requires a comparative analysis of all listed models. For each aspect in the list above, the AI must compare the models against each other based on real user reports and public discussions — not abstract specifications. The comparison must be structured by aspect, not by model — meaning hardware is covered across all models first, then software across all models, and so on. A final summary recommendation of which model is the better choice and why must be included, taking the country into account if specified. If independent reviews are insufficient for some models — the AI must explicitly state this.
